# 複数アプリ相互監視（ウォッチドッグ）設計調査

## このパターンはメジャーか？

**非常にメジャー**。以下の領域で広く使われている。

- 組み込み・産業システム（ハードウェアウォッチドッグが標準）
- POS・キオスク端末（常時稼働が求められる無人機器）
- Windows サービス（SCM の自動再起動機能が代表例）
- ゲームサーバー・ストリーミングサービスのデーモン管理
- Kubernetes の liveness probe / restart policy もこの概念の延長

「アプリが落ちたら誰かが検知して再起動する」は **信頼性工学の基本パターン**。

---

## 前提整理

| 項目 | 内容 |
|------|------|
| 実装言語 | **C#（.NET 8）** |
| 監視対象 | **.NET MAUI 製 Windows アプリ**（Delphi からの書き換え） |
| 対象 PC | **多数台**（店舗端末など） |
| 要件 | 管理サーバから各端末へのプロビジョニングが望ましい |

## ウォッチドッグ本体の設計（方式選択）

### 方式 A：単一ウォッチドッグ C# Windows サービス【推奨ベース】

```
[KcWatchdogService（C# Windows サービス）]
   ├─ 監視 → [AppA.exe]   落ちていたら Process.Start()
   ├─ 監視 → [AppB.exe]
   └─ 監視 → [AppC.exe]
```

**C# 実装（.NET 8 BackgroundService）**

```csharp
// dotnet new worker -n KcWatchdog
public class WatchdogWorker(ILogger<WatchdogWorker> logger, IConfiguration config)
    : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        var targets = config.GetSection("Targets").Get<TargetApp[]>()!;
        var timer = new PeriodicTimer(TimeSpan.FromSeconds(10));

        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            foreach (var t in targets)
            {
                if (Process.GetProcessesByName(t.ProcessName).Length == 0)
                {
                    logger.LogWarning("{Name} が停止しています。再起動します。", t.ProcessName);
                    Process.Start(new ProcessStartInfo(t.ExePath) { UseShellExecute = true });
                }
            }
        }
    }
}
```

**appsettings.json（監視対象の設定ファイル）**

```json
{
  "Targets": [
    { "ProcessName": "NeccaClient",  "ExePath": "C:\\Apps\\NeccaClient\\NeccaClient.exe" },
    { "ProcessName": "NM2006",       "ExePath": "C:\\Apps\\NM2006\\NM2006.exe" }
  ]
}
```

**Windows サービスとして登録（Program.cs）**

```csharp
var builder = Host.CreateApplicationBuilder(args);
builder.Services.AddWindowsService(opt => opt.ServiceName = "KcWatchdog");
builder.Services.AddHostedService<WatchdogWorker>();
var host = builder.Build();
host.Run();
```

**長所**
- 監視対象アプリの改修不要（プロセス名を appsettings.json に書くだけ）
- SCM 経由で OS 起動時に自動起動、SCM の障害回復で自身も自動再起動（SPOF 解消）
- appsettings.json を差し替えるだけで監視対象を変更可能

**短所**
- ハング検出はできない（プロセス消滅のみ）

---

### 方式 B：Named Pipe ハートビート（ハング検出が必要な場合）

各アプリが ping に応答するスレッドを持ち、ウォッチドッグが応答タイムアウトでハング判断。  
監視対象が .NET MAUI 製のため、**追加のライブラリなしで実装可能**。方式 A で運用開始後、必要になったら追加する。

> **監視対象が .NET MAUI である利点**  
> - Named Mutex / Named Pipe どちらも .NET 標準ライブラリで直接実装できる  
> - ウォッチドッグと同じ .NET エコシステムなので IPC のコード共有が容易  
> - NuGet パッケージを共有したハートビートライブラリを作ることもできる

## デプロイ・プロビジョニング方式

### 比較表

| 方式 | 追加ソフト | 初期設定コスト | 向いている場面 |
|------|-----------|--------------|--------------|
| P1: PowerShell Remoting（WinRM） | なし（OS 標準） | 低〜中 | LAN 内、ドメイン不要、無料 |
| P2: プル型自動更新（共有フォルダ / HTTP） | なし | 低 | 管理サーバが SMB 共有または HTTP を立てられる場合 |
| P3: Ansible for Windows | 管理サーバのみに Ansible | 中 | Linux 管理サーバがある場合、複数サイト管理 |
| P4: PDQ Deploy | 管理サーバのみに PDQ | 低（GUI 操作） | AD ドメインなし OK、SMB 環境 |
| P5: GPO + MSI | AD ドメイン必須 | 高 | 企業ドメイン環境 |

### P1：PowerShell Remoting（WinRM）【最も手軽・追加コストゼロ】

**概要**  
Windows に標準内蔵の WinRM（Windows Remote Management）を使い、管理サーバから各端末へ  
PowerShell スクリプトをリモート実行。ファイルコピー、サービス登録・起動まで一括自動化できる。

**端末側の事前設定（1 回だけ）**

```powershell
# 各端末で管理者として実行（または GPO / Ansible で一括適用可能）
Enable-PSRemoting -Force
Set-Item WSMan:\localhost\Client\TrustedHosts -Value "管理サーバのIP" -Force
```

**管理サーバ側のデプロイスクリプト例**

```powershell
# deploy_watchdog.ps1
$targets = @("192.168.1.101", "192.168.1.102", "192.168.1.103")  # 端末リスト
$cred    = Get-Credential   # 管理者アカウント

foreach ($pc in $targets) {
    $session = New-PSSession -ComputerName $pc -Credential $cred

    # 1. exe と設定ファイルをコピー
    Copy-Item -Path "\\管理サーバ\deploy\KcWatchdog\*" `
              -Destination "C:\Apps\KcWatchdog\" `
              -ToSession $session -Recurse -Force

    # 2. サービス登録（未登録の場合のみ）
    Invoke-Command -Session $session {
        $svc = Get-Service "KcWatchdog" -ErrorAction SilentlyContinue
        if (-not $svc) {
            sc.exe create KcWatchdog binPath= "C:\Apps\KcWatchdog\KcWatchdog.exe" start= auto
            sc.exe failure KcWatchdog reset= 60 actions= restart/5000/restart/10000/restart/30000
        }
        Start-Service "KcWatchdog" -ErrorAction SilentlyContinue
    }

    Remove-PSSession $session
    Write-Host "$pc : 完了"
}
```

**バージョンアップ時（再配布）**

```powershell
# 設定ファイルだけ差し替えて再起動
Invoke-Command -ComputerName $targets -Credential $cred {
    Stop-Service KcWatchdog
    Copy-Item "\\管理サーバ\deploy\KcWatchdog\appsettings.json" `
              "C:\Apps\KcWatchdog\appsettings.json" -Force
    Start-Service KcWatchdog
}
```

**評価**
- 追加ソフト不要
- スクリプト 1 本で何台でもデプロイ可能
- ドメイン不参加でも使える（ワークグループ OK、ただし TrustedHosts の設定が必要）

### P2：プル型自動更新（ウォッチドッグが自分自身を更新する）

**概要**  
ウォッチドッグ自身が起動時または定期的に管理サーバの共有フォルダ / HTTP エンドポイントを参照し、  
新バージョンがあれば自分を更新して再起動する。**各端末から来てもらう方式**。

```
[管理サーバ]
  共有フォルダ: \\server\deploy\KcWatchdog\
    ├─ version.txt     （例: "1.2.0"）
    ├─ KcWatchdog.exe
    └─ appsettings.json

[端末 N]
  KcWatchdogService
    起動時に version.txt を確認
    → 自分のバージョンと違えば exe と設定をコピーして自己再起動
```

**C# 実装イメージ（更新チェック部分）**

```csharp
public static async Task CheckAndUpdateAsync(string serverShare)
{
    var remoteVersion = await File.ReadAllTextAsync(Path.Combine(serverShare, "version.txt"));
    var localVersion  = Assembly.GetExecutingAssembly().GetName().Version!.ToString();

    if (remoteVersion.Trim() == localVersion) return;

    // 新しいバイナリをステージングにコピー
    File.Copy(Path.Combine(serverShare, "KcWatchdog.exe"),
              @"C:\Apps\KcWatchdog\KcWatchdog_new.exe", overwrite: true);

    // バッチで自己置換 & サービス再起動（遅延実行）
    File.WriteAllText(@"C:\Apps\KcWatchdog\update.bat", """
        timeout /t 2
        sc stop KcWatchdog
        copy /y KcWatchdog_new.exe KcWatchdog.exe
        sc start KcWatchdog
        del update.bat
        """);
    Process.Start("cmd.exe", "/c C:\\Apps\\KcWatchdog\\update.bat");
}
```

**評価**
- 各端末に管理者でログインする作業が一切不要
- 管理サーバ側で version.txt を更新するだけで全端末が自動更新
- 新端末追加時も「exe をコピーしてサービス登録する」1 回の作業だけ

### P3：Ansible for Windows

**概要**  
管理サーバに Ansible をインストールし、YAML で書いたプレイブックを実行。  
内部は WinRM を使うため端末側に追加ソフト不要。

**インベントリファイル（hosts.yml）**

```yaml
all:
  hosts:
    terminal01:
      ansible_host: 192.168.1.101
    terminal02:
      ansible_host: 192.168.1.102
  vars:
    ansible_user: Administrator
    ansible_password: "{{ vault_password }}"
    ansible_connection: winrm
    ansible_winrm_transport: ntlm
```

**プレイブック例（deploy_watchdog.yml）**

```yaml
- name: KcWatchdog デプロイ
  hosts: all
  tasks:
    - name: ディレクトリ作成
      ansible.windows.win_file:
        path: C:\Apps\KcWatchdog
        state: directory

    - name: exe コピー
      ansible.windows.win_copy:
        src: ./KcWatchdog.exe
        dest: C:\Apps\KcWatchdog\KcWatchdog.exe

    - name: 設定ファイルコピー
      ansible.windows.win_copy:
        src: ./appsettings.json
        dest: C:\Apps\KcWatchdog\appsettings.json

    - name: サービス登録・起動
      ansible.windows.win_service:
        name: KcWatchdog
        path: C:\Apps\KcWatchdog\KcWatchdog.exe
        start_mode: auto
        state: started
```

**実行**

```bash
ansible-playbook -i hosts.yml deploy_watchdog.yml
```

**評価**
- 冪等性（何度実行しても安全）
- 台数が増えても hosts.yml に追加するだけ
- 管理サーバが Linux でも Windows でも動く
- 学習コストあり（YAML プレイブックの習得が必要）

### P4：PDQ Deploy【GUI 操作・AD 不要・SMB 環境向け】

**概要**  
PDQ Deploy（Windows 専用ソフト）を管理サーバにインストール。  
GUI 操作でファイルコピー・コマンド実行・スケジュール配布が可能。  
Free 版でも基本的なデプロイは可能。

**手順概要**
1. 管理サーバに PDQ Deploy をインストール
2. デプロイパッケージを作成（コピー → sc create → sc start の順にステップ定義）
3. 端末リストを登録して「今すぐ展開」または「スケジュール展開」

**評価**
- GUI で直感的に操作できる
- AD ドメイン不要（ローカル管理者認証でも可）
- Free 版は同時配布数に制限あり

## 推奨構成（総合）

### 新規導入・中規模台数（〜50 台程度）

```
ウォッチドッグ本体:  方式 A（C# Windows サービス、.NET 8 self-contained single-file）
初回デプロイ:        P1（PowerShell Remoting）または P4（PDQ Deploy）
バージョンアップ:    P2（プル型自動更新）を組み込む

→ 初回は PowerShell スクリプト or PDQ で一括登録
→ 以後はウォッチドッグ自身が更新をプル。管理者作業ゼロ。
```

### 台数が多い・複数拠点・将来的な自動化投資あり

```
ウォッチドッグ本体:  方式 A（C# Windows サービス）
デプロイ・更新:      P3（Ansible）

→ hosts.yml に端末を追加するだけで新端末への配布が完結
→ プレイブックを CI/CD に組み込むと git push でデプロイ自動化も可能
```

## .NET self-contained single-file ビルド（配布を簡単にする）

監視対象の .NET MAUI アプリは Windows Desktop ランタイムをすでに必要とするが、  
ウォッチドッグサービスは独立してランタイムを含んだ single-file exe にしておくと  
デプロイ順序に依存せず単独で動作できる。

```bash
dotnet publish -c Release -r win-x64 \
  --self-contained true \
  -p:PublishSingleFile=true \
  -p:EnableCompressionInSingleFile=true
```

→ `KcWatchdog.exe` 1 ファイルだけ配布すれば動く（ランタイム込み、約 60〜80 MB）。  
→ `appsettings.json` と合わせて 2 ファイル配布で完結。

## サービス登録コマンド（1 端末での手動登録手順）

```powershell
# 管理者 PowerShell
$binPath = "C:\Apps\KcWatchdog\KcWatchdog.exe"
New-Service -Name "KcWatchdog" -BinaryPathName $binPath `
            -DisplayName "KC Watchdog Service" -StartupType Automatic
sc.exe failure KcWatchdog reset= 60 actions= restart/5000/restart/10000/restart/30000
Start-Service KcWatchdog
```

削除する場合：
```powershell
Stop-Service KcWatchdog
Remove-Service KcWatchdog   # Windows 10 1809 以降
```
